## 3. Clarify Deep-Dive: Why Mini (2/5) Is Worse Than Nano (3/5)

### Question Count Comparison

| Test Case | Expected Qs | Nano Qs | Mini Qs | Nano Pass | Mini Pass |
|-----------|-------------|---------|---------|-----------|-----------|
| chicken/rice/soy sauce (obvious) | 1 | 2 | 1 | FAIL | PASS |
| tortillas/kimchi/naan (mixed) | 2 | 2 | 2 | PASS | FAIL |
| chips/soda/cups (obvious party) | 1 | 1 | 2 | PASS | FAIL |
| flour/sugar/butter/eggs (baking) | 1 | 3 | 2 | FAIL | FAIL |
| salmon/steak/shrimp (ambiguous) | 2-3 | 3 | 3 | PASS | PASS |

### Mini's specific problems:

1. **Over-asks on obvious lists.** When the list clearly signals a party (chips/soda/cups/napkins), Mini still asks 2 questions including "Is this for a party, a snack restock, or something else?" -- the answer is obvious from the list. Nano correctly asks just 1 question.

2. **Generic questions that ignore list signals.** On the mixed-cuisine list (tortillas/kimchi/naan), Mini asks "What are you making?" and "How many people?" -- both generic. Nano asks cuisine-targeted questions with options like "Tacos or wraps", "Korean-style bowls", "Indian-style meal" that directly address the mixed signals.

3. **Options don't reference the actual items.** On the baking list, Mini's options are "Baking dessert", "Breakfast/brunch", "Just restocking", "Not sure yet" -- generic categories. Nano's options reference specific baked goods: "Cookies or bars", "Cake or cupcakes", "Baking bread".

### Side-by-side example -- chips/soda/cups/napkins:

**Nano (PASS, score=0.95):** 1 question, context-aware
```json
{"q": "What's this grocery run for?", "options": ["Game night snacks", "Weekend restock", "Small party", "One-time recipe"]}
```

**Mini (FAIL, score=0.55):** 2 questions, redundant
```json
{"q": "Is this for a party, a snack restock, or something else?", "options": ["Party", "Snack restock", "Weekly groceries", "Something else"]}
{"q": "How many people are you shopping for?", "options": ["Just me", "2-4 people", "5-8 people", "A crowd"]}
```

Mini asks a nearly self-answering question (cups + napkins = obviously not "just me") then adds a second question about headcount that could be combined.

### Summary: Mini over-generates questions and defaults to generic templates rather than reading list context. Nano better adapts question count and option text to the specific items.

## 5. Improvement Suggestions

### High Priority (fixes the most failures)

1. **Add explicit dietary constraint enforcement to the alternatives prompt.** Add a line like: *"If dietary restrictions are provided, EVERY alternative MUST comply. Never suggest an item that might contain the restricted ingredient. When uncertain, exclude it."* This fixes 4 failures across both models.

2. **Add conflict-acknowledgment instruction to inspire_item prompt.** Add: *"If the requested item conflicts with the user's dietary profile (e.g., meat item + vegetarian profile), explicitly acknowledge the conflict in a 'note' field before providing alternatives."* Fixes 2 failures.

3. **Add question-count guidance to the clarify prompt.** Add: *"For obvious lists (clear single purpose like baking or party), ask only 1 targeted question. For ambiguous lists (mixed signals, unclear scale), ask 2-3 questions. Never ask more than 3."* Fixes 4 failures.

### Medium Priority

4. **Add match-level calibration to alternatives prompt.** Add: *"'Very close' means nearly identical (e.g., one brand of sriracha for another). Plant-based substitutes for meat should be 'Similar' at best."* Fixes 2 failures.

5. **Require clarify options to reference list items.** Add: *"Options should reference specific items or combinations from the user's list, not generic categories."* Improves Mini's clarify quality.

### Low Priority

6. **Add disambiguation requirement to translate prompt.** Add: *"If the item has a common confusion pair in the target language (e.g., baking soda vs baking powder), include a warning note."* Fixes 2 failures.

7. **Standardize category naming in item_info prompt.** Provide example categories: "produce", "dairy/refrigerated", "grains/dry goods", "spices/seasonings", "condiments/sauces" so the model matches expected taxonomy.

### Model Recommendation

**Use Nano as the default model for all sync endpoints.** It scores equal or better than Mini on 4/5 endpoints and costs 3.75x less. Mini shows no clear advantage on any endpoint. Consider Mini only if future complex endpoints (suggest, per-list inspire) benefit from it.
